# InterWise 用户开放前推进计划

## Summary
当前目标不是立刻大规模商用，而是把已部署到 Azure VM 的 InterWise 从“能访问”推进到“可信用户可内测、问题可追踪、成本可控制、数据可恢复”。优先级按风险排序：先保护成本和接口，再补产品入口和反馈闭环，然后做轻量监控与备份。

## Phase 1：产品入口与用户反馈
- 新增未登录可访问的产品介绍页，展示 InterWise 的定位、核心功能、适合人群、使用步骤和当前内测限制。
- 登录前首页偏“官网/说明页”，登录后进入现有工作台。
- 在首页和应用内加入反馈入口，支持用户提交问题类型、反馈内容、联系方式和当前页面。
- 增加基础使用说明：如何注册、上传简历、开始文字面试、查看报告、使用 AI Mentor。
- 明确提示：当前 HTTP 公网 IP 阶段优先开放文字面试；视频面试需 HTTPS 后再开放。

## Phase 2：访问统计与行为记录
- 增加轻量访问/事件统计，不先做复杂运营后台。
- 记录关键事件：访问首页、注册成功、登录成功、开始面试、结束面试、上传简历、查看报告、刷新 Mentor、提交反馈。
- 后端记录关键异常事件：验证码失败、DeepSeek 调用失败、SSE 中断、文件上传失败、接口 500。
- 先通过数据库查询或简单统计页查看 PV、UV、注册数、面试开始数、面试完成率、反馈数量。
- 统计数据禁止保存密码、token、简历正文、完整面试对话等敏感内容。

## Phase 3：限流、防刷与成本保护
- 给公开接口加限流：登录、注册、验证码、重置密码、开始面试、简历上传、Mentor 刷新。
- 未登录接口按 IP 限流，登录后接口按用户 ID + IP 限流。
- 给高成本 AI 功能加每日额度，例如每用户每日面试次数、Mentor 刷新次数、简历解析次数。
- 验证码保留邮箱 60 秒限制，并补充 IP 级限制，防止 SMTP 被刷。
- 超限时返回友好提示，例如“请求过于频繁，请稍后再试”，不暴露内部规则。

## Phase 4：轻量监控与告警
- 监控应运行在 Azure、服务器或外部服务上，不依赖本地电脑开机。
- 使用 Azure Monitor 监控 VM 的 CPU、内存、磁盘、网络和预算。
- 使用 UptimeRobot 或类似服务从外网定时访问网站，网站不可访问时邮件提醒。
- 在服务器上用 cron 定时检查 Docker 容器状态、首页访问、后端健康接口和磁盘空间。
- 后端增加健康检查接口，至少覆盖：后端存活、MySQL、Redis、Qdrant。
- 第一版不引入 Prometheus/Grafana/ELK，避免运维复杂度过高。

## Phase 5：备份自动化
- 每天自动备份 MySQL。
- 每天或每周备份上传文件目录 `uploads`。
- Qdrant 数据可定期备份；如果索引可从 MySQL 题库重建，优先保护 MySQL 和上传文件。
- 备份不能只放在同一台 VM，至少定期下载到本地或上传到对象存储。
- 每月做一次恢复演练，确认备份真的能恢复。

## Phase 6：HTTPS 与视频面试开放
- 购买域名并解析到 Azure 公网 IP。
- 开放 443，配置 HTTPS 证书。
- 将 CORS、MCP origin 等配置从公网 IP 切换到 HTTPS 域名。
- 完成 HTTPS 后再开放视频面试、麦克风、摄像头和情绪分析。
- 视频/情绪分析页面增加明确授权说明。

## Phase 7：隐私与用户信任
- 增加隐私说明页面，说明邮箱、简历、面试记录、AI 评估、头像、视频/情绪数据的用途。
- 增加用户数据删除能力的最小版本：删除简历画像、删除面试记录、删除头像。
- 后续再考虑账号注销、数据导出、用户协议等完整合规能力。

## Recommended Order
1. 产品介绍页 + 反馈入口。
2. 访问统计和关键事件记录。
3. 限流、防刷、AI 调用额度。
4. 健康检查 + Azure Monitor + 外部可用性监控。
5. MySQL/uploads 自动备份。
6. 域名 + HTTPS。
7. 视频面试正式开放。
8. 简单运营统计页和后续 CI/CD。

## Acceptance Criteria
- 用户未登录也能理解产品是什么、怎么用、如何反馈。
- 你能看到每日访问量、注册数、面试开始数、面试完成数和反馈数量。
- 高频请求不会无限消耗验证码、LLM 和服务器资源。
- 网站挂掉、容器异常、磁盘快满时你能收到提醒。
- 数据库或上传文件出问题时有可用备份。
- 视频面试只在 HTTPS 环境下开放。
